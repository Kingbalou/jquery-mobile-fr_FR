Liens API

jQuery Mobile est conçu pour fonctionner avec les conventions de liaison d'une page standard et des couches de navigation AJAX en plus pour une compatibilité maximale

Liaison de pages

Vous pouvez lier des pages et des ressources comme vous le feriez normalement. jQuery Mobile gérera automatiquement les demandes de page dans un modèle de page unique, utilisant Ajax, si cela est possible. Lorsqu'Ajax n'est pas possible (comme une url qui n'est pas dans le même domaine ou si cela est spécifié en utilisant certains attributs sur le lien), une demande http normale est utilisée à la place.

L'objectif de ce modèle est de permettre aux développeurs de créer des sites Web en utilisant les meilleures pratiques - où les liens ordinaires "fonctionneront" sans aucune configuration spéciale - tout en créant une expérience riche qui ne peut pas être réalisée avec des requêtes HTTP standard.

Le comportement par défaut du lien : AJAX

Afin de permettre des transitions de page animée, tous les liens qui pointent vers des pages externes (ex. products.html) seront chargées via AJAX. Pour le faire discrètement, le framework analyse le href du lien pour formuler une requête Ajax (Hijax) et affiche l'icône de chargement. Tout cela se fait automatiquement par jQuery Mobile.

Si la requête Ajax est réussie, le contenu de la page est ajoutée au DOM, tous les widgets mobiles sont auto-initialisés et la nouvelle page est animée dans la vue avec une transition de page.

Si la requête Ajax échoue, le framework affiche en superposition un petit message d'erreur (stylisé avec le nuancier "e") qui disparaît après un bref instant afin que ceci ne casse pas le flux de la navigation. Aperçu d'un exemple de message d'erreur.

Remarque : vous ne pouvez pas faire un lien à un document multipage avec la navigation AJAX active, car le framework ne fera que charger la première page qu'il trouve, et non l'ensemble des pages internes. Dans ces cas, vous devrez lier sans Ajax (voir la prochaine section) pour actualiser la totalité de la page afin d'empêcher d'éventuelles collisions de hash. Il y a actuellement un plugin subpage qui permet de charger des documents multi-pages.

Liaison sans AJAX

Les liens qui pointent vers d'autres domaines ou qui ont les attributs rel="external", data-ajax="false" ou target ne seront pas chargés avec l'AJAX. Au lieu de cela, ces liens provoqueront un rafraîchissement de la page complète sans transition animée. Les deux attributs (rel="external" et data-ajax="false") ont le même effet, mais ils ont un sens sémantique différent : rel="external" doit être utilisé lors de la liaison vers un autre site ou un autre domaine, alors que data-ajax="false" est utile pour tout simplement empêcher une page dans votre domaine d'être chargée via AJAX. En raison des restrictions de sécurité, le framework choisit toujours pour les liens vers des domaines externes le comportement sans l'AJAX.

Cela évite le besoin d'ajouter cet attribut pour chaque lien dans un conteneur. Remarque : Lors de la construction d'une application jQuery Mobile, où le système de navigation Ajax sera désactivé globalement ou fréquemment sur des liens individuels, nous vous recommandons de désactiver l'option de configuration globale $.mobile.pushStateEnabled pour éviter les comportements de navigation incompatibles dans certains navigateurs.

Liaison dans un document multi-page

Un unique document HTML peut contenir un ou plusieurs conteneurs de "page" en empilant simplement plusieurs divs avec le data-role de "page". Cela vous permet de construire un petit site ou une application dans un seul document HTML, jQuery Mobile affichera, lorsque la page se chargera, simplement la première "page" qu'il trouvera dans l'ordre du code source.

Si un lien dans un document multi-page pointe vers une ancre (# foo), le framework regardera vers l'enveloppe de la page avec cet ID (id="foo"). S'il trouve une page dans le document HTML, il fera la transition vers la nouvelle page. Vous pouvez facilement naviguer entre les "pages" locales, internes et les pages externes dans jQuery Mobile. Les deux auront le même aspect pour l'utilisateur final, sauf que les pages externes afficheront l'icône AJAX pendant le chargement. Dans les deux cas, jQuery mobile met à jour le hash de l'URL de la page pour activer le support du bouton retour, du lien profond et du marque-page.

Il est important de noter que si vous avez une liaison d'une page mobile, qui a été chargée via AJAX, à une page qui contient plusieurs pages internes, vous devez ajouter un rel="external" ou un data-ajax="false" au lien. Ceci indique au framework de faire un rechargement de la page complète pour vider le hash AJAX dans l'URL. Cela est essentiel car les pages Ajax utilisent le hash (#) pour suivre l'historique d'AJAX, alors que plusieurs pages internes utilisent le hash pour indiquer les pages internes donc il y aura des conflits dans le hash entre ces deux modes.

Par exemple, un lien vers une page contenant plusieurs pages internes devrait ressembler à ceci :

<a href="multipage.html" rel="external">Lien Multi-page</a>

Liens du bouton "Retour"

Si vous utilisez l'attribut data-rel="back" sur une ancre, tout clic sur cette ancre imitera le bouton retour, remontant une entrée de l'historique et en ignorant le href par défaut de l'ancre. Ceci est particulièrement utile lors de la génération des boutons "back" avec JavaScript, de la même manière qu'un bouton pour fermer une boîte de dialogue.

Lorsque vous utilisez cette fonctionnalité dans votre balisage source, bien que les navigateurs qui supportent cette fonctionnalité n'utiliseront pas l'attribut href spécifié, assurez vous de toujours fournir une valeur significative qui pointe actuellement vers l'URL de la page référente afin de permettre à la fonctionnalité de fonctionner pour les utilisateurs de navigateurs de Grade C. Si les utilisateurs peuvent atteindre cette page depuis plus d'une page référante, spécifiez un href judicieux de sorte que la navigation reste logique pour tous les utilisateurs.

Aussi, s'il vous plaît, gardez à l'esprit que si vous voulez juste une transition inverse sans réellement retourner dans l'historique, vous devez utiliser à la place l'attribut data-direction="reverse". Remarque : data-direction="reverse" est destiné à exécuter simplement la version inverse de la transition qui se déroulera sur ce changement de page, tandis que data-rel="back" fait le lien fonctionnellement équivalent à un bouton retour du navigateur et toutes les logiques standard du bouton de retour s'applique. L'ajout de data-direction="reverse" à un lien avec data-rel="back" ne sera pas l'inverse de la transition de page inversée et produira une version "normale" de la transition.

Les redirections et les liens vers les répertoires

Lors de la liaison à l'index du répertoire (comme href="typesofcats/" à la place de href="typesofcats/index.html"), vous devez fournir un slash. C'est parce que jQuery Mobile suppose que la section après le dernier caractère "/" dans une URL est un nom de fichier, et il va supprimer cette section lors de la création des urls de base à partir duquel les futures pages seront référencées.

Cependant, vous pouvez contourner ce problème en retournant votre div de page avec un attribut déjà spécifié data-url. Quand vous faites cela, jQuery Mobile utilisera la valeur de l'attribut en mettant à jour l'URL, au lieu de l'URL utilisée pour requêter la page. Cela vous permet également de retourner les URLs qui changent à la suite d'une redirection, par exemple, vous pourriez afficher un formulaire vers "/login.html" mais renvoyer une page vers l'URL "/compte" après une soumission réussie. Cet outil vous permet dans ces situations de prendre le contrôle de la pile de l'historique de jQuery Mobile.

Remarques sur les liens

L'environnement non-standard créé par le modèle de navigation de page de jQuery Mobile introduit quelques conditions pour lesquelles vous devez être conscient lors de la construction des pages :

  • Lors de la liaison aux répertoires, sans un nom de fichier dans l'URL, (comme href="typesofcats/" à la place de href="typesofcats/index.html"), vous devez fournir un slash. C'est parce que jQuery Mobile suppose que la section après le dernier caractère "/" dans une URL est un nom de fichier, et il va supprimer cette section lors de la création des URLs de base à partir duquel les futures pages seront référencées.

  • Les documents chargés via AJAX sélectionneront la première page dans le DOM de ce document qui sera chargée comme un élément de page jQuery Mobile. En conséquence, le développeur doit s'assurer de la gestion des attributs id de la page chargée et des éléments enfants pour éviter toute confusion lors de la manipulation du DOM.

  • Si vous accédez à un document multi-page, vous devez utiliser un attribut data-ajax="false" sur le lien pour provoquer une actualisation complète de la page dû à la limitation ci-dessus où nous ne chargeons que le nœud de la première page dans une requête AJAX en raison du risque de collisions du hash. Il y a actuellement un plugin subpage qui permet de charger des documents multi-page.

  • Lorsque vous liez les pages intérieures d'un modèle multi-pages, vous ne devriez pas utiliser l'attribut data-ajax="false" car il est d'aucune utilité et il interférera avec les paramètres de transition.

  • Le nom de la clé de "ui-page" utilisé dans les références des sous-hash d'URL peut être réglé à la valeur que vous souhaitez, afin de se fondre dans la structure de votre URL. Cette valeur est stockée dans jQuery.mobile.subPageUrlKey.

  • Lorsque vous navigez vers un document jQuery Mobile précédemment chargé depuis un externe ou d'un document interne avec le plugin pushState activé, certains navigateurs chargent et déclenchent l'événement popstate sur le mauvaix document ou pour de mauvaises raisons (deux effets de bords enregistrés à ce jour). Si vous avez régulièrement des liens vers des documents externes et vous trouvezq que l'application se comporte de façon erratique, essayez de désactiver le support pushstate.

  • jQuery Mobile ne prend pas en charge le paramètre de requête passé vers des pages internes/embarquées mais il y a deux plugins que vous pouvez ajouter à votre projet pour supporter cette fonctionnalité. Il en existe un léger page params plugin et un plus complet jQuery Mobile router plugin pour utiliser backbone.js ou spine.js. Un nouveau plugin appelé routerlite simplifie cela avec seulement quatre méthodes : routeinit, routechange, pageinit et pagechange.

  • Certaines applications externes (notamment la mise en œuvre de Facebook OAuth) modifient leur URL de réponse de manière que cela interfère avec jQuery Mobile. En particulier, Facebook ajoute #_=_ à la fin du la fonction de rappel. Actuellement, la meilleure solution pour cela est de le retirer de la table de hach avant que jQuery Mobile le charge en utilisant quelque chose comme : if (window.location.hash == "#_=_") window.location.hash = ""; . jQuery Mobile peut ensuite traiter et améliorer la page correctement.
