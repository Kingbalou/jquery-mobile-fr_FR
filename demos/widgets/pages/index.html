Pages API

La page est l'unité primaire de l'interaction dans jQuery Mobile et elle est utilisée pour grouper le contenu dans des vues logiques qui peuvent être animées dans et hors de la vue avec des transitions de page. Un document HTML peut commencer avec une seule "page" et le système de navigation AJAX va charger des pages supplémentaires à la demande dans les DOM lorsque les utilisateurs naviguent. Alternativement, un document HTML peut être construit avec plusieurs «pages» à l'intérieur et le framework fera la transition entre ces vue locales sans avoir besoin de demander le contenu au serveur.

La structure d'une page mobile

Un site jQuery Mobile doit commencer par un doctype HTML5 pour profiter pleinement de toutes les fonctionnalités du framework. (Les anciens appareils possédant des navigateurs qui ne comprennent pas le HTML5 vont ignorer le 'doctype' et divers attributs personnalisés.)

Dans le head, les références pour jQuery, jQuery Mobile et le CSS du thème mobile sont tous nécessaires pour démarrer les choses. jQuery Mobile 1.3.0 fonctionne avec les versions jQuery de la 1.7.0 à la 1.9.1. La meilleure façon de commencer est de mettre un lien vers les fichiers hébergés sur le CDN de jQuery ou pour de meilleures performances, construire un lot personnalisé :


<!DOCTYPE html> 
<html>
<head>
	<title>Titre de la page</title>
	<meta name="viewport" content="width=device-width, initial-scale=1">
	<link rel="stylesheet" href="http://code.jquery.com/mobile/1.3.0/jquery.mobile-1.3.0.min.css" />
	<script src="http://code.jquery.com/jquery-1.9.1.min.js"></script>
	<script src="http://code.jquery.com/mobile/1.3.0/jquery.mobile-1.3.0.min.js"></script>
</head>

<body>
	...le contenu se place ici...
</body>
</html>

La balise META viewport

Notez ci-dessus qu'il y a une balise META viewport dans le head pour préciser comment le navigateur doit afficher le niveau de zoom de la page et les dimensions. Si ce n'est pas défini, de nombreux navigateurs mobiles vont utiliser une largeur « virtuelle » de la page autour de 900 pixels pour lui permettre de bien travailler avec les sites existants de bureau, mais les écrans peuvent avoir un look dézoomé et trop large. En définissant les attributs de viewport à content="width=device-width, initial-scale=1", la largeur sera mise à celle de la largeur en pixels de l'écran de l'appareil. 


<meta name="viewport" content="width=device-width, initial-scale=1">

Ces paramètres ne désactivent pas la capacité de l'utilisateur de zoomer sur les pages, ce qui est agréable dans une perspective d'accessibilité. Il y a un problème mineur dans iOS qui ne définit pas correctement la largeur lors d'un changement d'orientation avec ces paramètres viewport, mais ce sera, espérons le, corrigé dans une future version. Vous pouvez définir d'autres valeurs viewport pour désactiver le zoom, si nécessaire, car cela fait partie du contenu de votre page et non de la bibliothèque.

A l'intérieur du body : les pages

A l'intérieur de la balise <body>, chaque vue ou "page" sur l'appareil mobile est identifié par un élément (généralement un div) avec l'attribut data-role="page". 


<div data-role="page">
	...
</div>

Au sein du conteneur de la "page", n'importe quel balisage HTML valide peut être utilisé, mais pour les pages typiques de jQuery Mobile, les enfants immédiats d'une « page » sont des divs avec des data-roles "header", "content" et "footer".


<div data-role="page">
	<div data-role="header">...</div>
	<div data-role="content">...</div>
	<div data-role="footer">...</div>
</div>

En assemblant le tout : c'est un modèle de page unique standard

En assemblant la totalité, c'est le modèle passe-partout standard de page, vous devez commencer avec un projet : 


<!DOCTYPE html>
<html>
<head>
	<title>Titre de la page</title>

	<meta name="viewport" content="width=device-width, initial-scale=1">

	<link rel="stylesheet" href="http://code.jquery.com/mobile/1.3.0/jquery.mobile-1.3.0.min.css" />
	<script src="http://code.jquery.com/jquery-1.9.1.min.js"></script>
	<script src="http://code.jquery.com/mobile/1.3.0/jquery.mobile-1.3.0.min.js"></script>
</head>
<body>

<div data-role="page">

	<div data-role="header">
		<h1>Titre de la page</h1>
	</div><!-- /header -->

	<div data-role="content">
		<p>Le contenu de la page se place ici.</p>
	</div><!-- /content -->

	<div data-role="footer">
		<h4>Pied de page</h4>
	</div><!-- /footer -->
</div><!-- /page -->

</body>
</html>
Modèle de page

La structure du modèle multi-page

Un unique document HTML peut contenir plusieurs "pages" qui sont chargées ensemble par l'empilage de divs avec le data-role de "page". Chaque bloc de "page" a besoin d'un ID unique (id="foo") qui sera utilisé pour relier en interne les "pages" entre elles (href="# foo"). Quand un lien est cliqué, le framework va chercher une "page" interne avec l'ID et la fait apparaitre via une transition.

Voici un exemple de deux "pages" du site construites avec deux divs jQuery Mobile navigables en liant un ID placé sur l'enveloppe de chaque page. Notez que les ids sur les conteneurs de page sont uniquement nécessaires pour supporter les liens internes des pages, et ils sont facultatifs si chaque page est un document HTML distinct. Voici à quoi correspond deux pages à l'intérieur de l'élément body.


<body>

<!-- Début de la première page -->
<div data-role="page" id="foo">

	<div data-role="header">
		<h1>Foo</h1>
	</div><!-- /header -->

	<div data-role="content">
		<p>e suis le premier dans l'ordre de la source donc je suis la page affichée.</p>
		<p>Voir la page interne appelée <a href="#bar">bar</a></p>
	</div><!-- /content -->

	<div data-role="footer">
		<h4>Pied de page</h4>
	</div><!-- /footer -->
</div><!-- /page -->

<!-- Début de la seconde page -->
<div data-role="page" id="bar">

	<div data-role="header">
		<h1>Bar</h1>
	</div><!-- /header -->

	<div data-role="content">
		<p>e suis le deuxième dans l'ordre de la source donc je suis caché lors du chargement de la page. Je suis juste montré, si un lien qui référence mon ID est cliqué.</p>
		<p><a href="#foo">Retour vers foo</a></p>
	</div><!-- /content -->

	<div data-role="footer">
		<h4>Pied de page</h4>
	</div><!-- /footer -->
</div><!-- /page -->
</body>
Voir un modèle multi-page

MERCI DE NOTER : Puisque nous utilisons le hash pour suivre l'historique de navigation pour toutes les "pages" AJAX, il n'est pas possible actuellement de faire un lien vers une ancre (index.html#foo) sur une page en jQuery Mobile. Car le framework va chercher une "page" avec un id #foo au lieu d'utiliser le comportement natif qui consiste à se déplacer vers le contenu de cet id.

L'attribut id de tous vos éléments doivent être non seulement unique sur une page donnée, mais aussi unique à travers les pages d'un site. C'est parce que le modèle de navigation d'une seule page de jQuery Mobile permet à de nombreuses "pages" différentes d'être présentes dans le DOM au même moment. Cette disposition s'applique également lorsque vous utilisez un modèle multi-page, puisque tous les "pages" sur le modèle sont chargés en une seule fois.

C'est des conventions, ce n'est pas des obligations

Bien que la structure de la page décrite au-dessus est une méthode recommandée pour la construction avec jQuery Mobile d'une application web standard, le framework est très flexible avec la structure du document. Les éléments data-role : page, header, content et footer sont facultatifs et ils sont surtout utiles pour fournir un formatage et une structure de base. L'enveloppement utilisé pour la page est obligatoire pour que l'auto-initialisation fonctionne, mais cela est désormais facultatif pour les documents de page unique donc il n'y a pas l'obligation de tout baliser. Pour une page web avec une mise en page personnalisée, l'ensemble de ces éléments structuraux peuvent être omis, la navigation AJAX et tous les widgets fonctionneront exactement comme ils le font dans la structure passe-partout. En arrière plan, le framework va injecter l'enveloppement de la page si elle n'est pas incluse dans le balisage, car c'est nécessaire pour gérer les pages, mais le balisage de départ peut maintenant être extrêmement simple.

Notez que dans une configuration multi-page, vous êtes tenus d'avoir l'enveloppement de page dans votre balisage afin de regrouper le contenu dans plusieurs pages.

Autre Note : Si votre body contient aucun divs data-role="page", jQuery Mobile enveloppe entièrement le contenu du body à l'intérieur d'un div de page comme expliqué au dessus. jQuery Mobile utilise la méthode de jQuery wrapAll() pour faire ceci qui cherche des balises de script à l'intérieur du contenu enveloppé et charge chaque source de script via XHR. Si les scripts sont présents dans le corps, le navigateur termine en les chargeant deux fois. Nous recommandons donc vivement que les documents jQuery Mobile avec des scripts dans leur body contiennent aussi un div avec data-role="page".

Préchargement des pages

Lors de l'utilisation des modèles d'une seule page, vous pouvez précharger les pages dans le DOM afin qu'ils soient disponibles instantanément lorsque l'utilisateur les visites. Pour précharger une page, ajouter l'attribut data-prefetch à un lien qui pointe vers la page. jQuery mobile charge ensuite la page cible en arrière plan après que la page principale soit chargée et que l'événement pagecreate soit déclenché.

Ce lien pré-chargera la page

Alternativement, vous pouvez précharger une page par programme en utilisant $.mobile.loadPage() :


$.mobile.loadPage( pageUrl, { showLoadMsg: false } );

Cache du DOM

Cependant, le maintien d'anciennes pages dans les DOM remplit rapidement la mémoire du navigateur, et peut, sur certains navigateurs mobiles, ralentir ou même planter. jQuery Mobile a un mécanisme simple pour garder le DOM rangé.

Chaque fois qu'il charge une page via Ajax, il marque la page pour qu'elle soit retirée du DOM lorsque vous en sortez (techniquement, sur l'événement pagehide). Si vous revisitez une page enlevée, le navigateur peut être en mesure de récupérer le fichier HTML de la page dans son cache. Si ce n'est pas le cas, il recharge le fichier depuis le serveur. (Dans le cas des vues de liste imbriquée, jQuery Mobile supprime toutes les pages qui composent la liste imbriquée une fois que vous accédez à une page qui ne fait pas partie de la liste.)

Si vous préférez, vous pouvez dire à jQuery Mobile de conserver les pages visitées précédemment dans le DOM au lieu de les éliminer. Cela vous permet de mettre en cache les pages afin qu'elles soient disponibles instantanément si l'utilisateur revient vers elles.


$.mobile.page.prototype.options.domCache = true;

Sinon, pour mettre en cache seulement une page particulière, vous pouvez ajouter l'attribut data-dom-cache="true" au conteneur de la page.

Pour garder toutes les pages visitées précédemment dans le DOM, définissez l'option domCache sur le plugin page à true comme ceci :


pageContainerElement.page({ domCache: true });

Notez que le contenu de la première page n'est pas supprimé du DOM, seules les pages chargées via AJAX. Les pages à l'intérieur d'un modèle multi-page ne sont pas affectées par cette fonction pour tout - jQuery Mobile supprime uniquement les pages chargées via AJAX.
