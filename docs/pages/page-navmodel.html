Modèle de navigation de jQuery Mobile

Une "page" dans jQuery Mobile est composée d'un élément (généralement un div ) avec un attribut data-role défini à "page", qui contient généralement des éléments div avec des rôles "header", "content" et "footer", chacun contenant le balisage commun, les formulaires et les widgets personnalisés de jQuery Mobile.

L'automatisation des processus de base pour un chargement d'une page est la suivante : d'abord, une page est appelée par une requête HTTP normale, et les "pages" suivantes sont alors appelées et injectées dans le DOM de cette page. Pour cette raison, le DOM peut avoir un certain nombre de «pages» en même temps, dont chacune peut être revisitée en liant son attribut data-url.

Quand une URL est initialement appelée, il peut y avoir un ou plusieurs "pages" dans la réponse et seule la première sera affichée. L'avantage de stocker plus d'une "page" est de permettre de pré-télécharger les pages statiques qui sont susceptibles d'être visitées.

Navigation des pages pilotée par hash et Ajax

Par défaut, toute la navigation dans jQuery Mobile est basée sur les changements et les mises à jour de location.hash. Autant que possible, les changements de page utiliseront une transition en douceur entre la "page" courante et la suivante, si elle est soit déjà présente dans les DOM ou soit automatiquement chargée via Ajax.

Les valeurs de hash créée par jQuery Mobile sont normalisées comme des chemins complets par rapport à l'URL de la première "vraie" page qui a été chargée. Le hash est toujours maintenue comme une URL valide, de sorte que toute «page» dans jQuery mobile puisse être marquée ou référencée dans un lien. Pour récupérer une URL non basée sur le hash, il suffit de retirer le # à partir de l'adresse et actualiser la page.

En général, les changements de hash sont créés chaque fois qu'un lien est cliqué dans jQuery Mobile. Quand un lien est cliqué, jQuery mobile s'assurera que le lien fait référence à une URL locale, et si c'est le cas, il va prévenir le comportement par défaut du clic du lien de se produire et demander l'URL référencées via Ajax à la place. Lorsque la page retourne avec succès, cela définira le location.hash avec l'url relative de la nouvelle page.

Lorsque la page retourne avec succès, cela définira à le location.hash avec l'url relative de la nouvelle page . Quand un changement de hash survient (et aussi quand la page se charge en premier), le gestionnaire d'événements hashchange enverra le location.hash à la fonction $.mobile.changePage() qui à son tour, soit charge ou soit révèle la page référencée.

Une fois que la page référencée est présente dans le DOM, la fonction $.mobile.changePage() applique une transition entre la page active actuelle et la nouvelle page. Les transitions de page se produisent grâce à l'ajout et la suppression des classes qui s'appliquent aux animations CSS. Par exemple, dans une transition slide-left, la page existante est donnée aux classes "slideleft" et "out", et la page entrante est donnée aux classes "slideleft" and "in", ainsi qu'à une classe "ui-page-active" pour la marquer comme la nouvelle page "active" à la visualisation. Lorsque l'animation est terminée, les classes "in" et "out" sont supprimées, et la page quittée perd sa classe "ui-page-active".

Plugin pushState

Il y a une fonctionnalité optionnelle qui convertit les plus longues URLs basées sur le hash mentionné dans la section précédente à l'intérieur du chemin complet du document qui est plus propre et rend le suivi de l'Ajax transparent dans la structure d'URL. C'est construit comme une amélioration au dessus du système d'URL basé sur le hash pour les liens Ajax. Notez que malgré le nom, cette fonctionnalité convertit techniquement les urls basées sur le hash en utilisant history.replaceState (c'est pas history.pushState) dans la version actuelle, car cela fonctionne de façon plus fiable à travers nos plateformes cibles. Pour les navigateurs qui ne supportent pas history.replaceState ou si cette fonctionnalité est désactivée, les URLs basées sur le hash seront utilisées à la place.

Le plugin initialise lorsque le DOM est complètement chargé, vous pouvez l'activer et le désactiver manuellement en définissant l'option de configuration globale $.mobile.pushStateEnabled à false n'importe quand avant que le document soit ready.

Important : rel="external" et $.mobile.ajaxEnabled=false

Les implémentations légèrement différentes de l'API replaceState dans différents navigateurs peuvent provoquer des comportements étranges dans des scénarios spécifiques. Par exemple, certaines implémentations de navigateur (y compris les navigateurs de bureau) mettent en œuvre l'événement popstate différemment lors de la liaison externe et reviennent à une page sur laquelle l'état a déjà été poussé/remplacé Lors de la construction d'une application jQuery Mobile où le système de navigation Ajax est explicitement désactivé, soit par l'utilisation fréquente de rel="external" sur des liens ou soit par la désactivation complète de la navigation Ajax via $.mobile.ajaxEnabled=false, nous vous recommandons la désactivation de la fonctionnalité pushState pour revenir à la navigation basée sur le hash pour un comportement plus cohérent.

changePage

Dans le framework, les changements de page - à la fois pour les pages déjà dans les DOM et pour les pages qui doivent être chargées via Ajax - utilisez la fonction $.mobile.changePage(). $.mobile.changePage() contient toute la logique pour trouver les pages de départ et d'arrivée pour la transition et comment gérer les diverses réponses tel que la réponse d'une page non trouvé. $.mobile.changePage() peut être appelé en externe et accepte les arguments suivants (to, transition, back, changeHash). L'argument to peut accepter soit une chaîne (comme une url de fichier ou l'ID de l'élément local), un tableau (dans lequel le premier élément du tableau est une page locale à partir de laquelle vous souhaitez faire la transition et le second élément du tableau est la page to) ou un objet (avec des propriétés attendues : url, type ("get" ou "post") et data (pour les paramètres sérialisés)), dont la dernière est utile pour le chargement de pages qui attendent des données de formulaire. L'argument transition accepte une chaîne représentant une transition nommé, comme "slide". L'argument back accepte un booléen représentant si la transition doit aller en avant ou en arrière. Enfin, l'argument changeHash accepte un booléen pour savoir si vous voulez que l'url soit mise à jour sur un changement de page réussie.

La fonction $.mobile.changePage() est utilisée dans un certain nombre d'endroit dans jQuery Mobile. Par exemple, quand un lien est cliqué, son attribut href est normalisé et ensuite $.mobile.changePage() s'occupe du reste. Lorsque les formulaires sont soumis, jQuery mobile rassemble simplement un peu d'attributs du formulaire, sérialise ses données, et encore une fois, $.mobile.changePage() est utilisé pour gérer la soumission et la réponse. En outre, des liens qui créent des boîtes de dialogue utilisent $.mobile.changePage()pour ouvrir une page référencée sans mettre à jour la table de hash, ce qui est utile pour garder les boîtes de dialogue en dehors du suivi de l'historique.

Élément de base

Un autre ingrédient clé du modèle de navigation de page de jQuery Mobile est l'élément base, qui est injecté dans le head et est modifié à chaque changement de page afin de s'assurer que tous les ressources (images, CSS, JS, etc) référencés sur cette page seront requêtés à partir du bon chemin. Dans les navigateurs qui ne supportent pas les mises à jour dynamiques pour l'élément base (comme Firefox 3.6), jQuery Mobile boucle par le biais de tous les ressources référencées sur la page et les préfixes de leurs attributs href et src avec le chemin de base.

Explication de développeur sur la gestion des URL de base :

jQuery Mobile gère les requêtes HTTP en utilisant une combinaison de chemins générés d'URL absolue et la manipulation d'un élément généré <base> de l'attribut href. La combinaison de ces deux approches nous permet de créer des URL qui contiennent des informations de chemin complet pour le chargement de page, et un élément de base pour bien orienter les requêtes des ressources faites par ces pages chargées (comme les images et les feuilles de style).

Stockage de data-url

Le modèle de navigation maintient un attribut d'URL de données sur toutes les éléments data-role="page". Cet attribut data-url est utilisé pour retracer l'origine de l'élément de la page. Les pages incorporées dans le document d'application principal ont toutes leur paramètre data-url définit avec l'ID de l'élément @data-role="page". La seule exception à cela est la première page dans le document. La première page-est spéciale car elle peut être adressée par son @id, si elle en a un, ou par le document ou l'URL de base (avec aucun fragment du hash).

Les pages qui sont externes au document d'application sont tirés dynamiquement via AJAX, leur data-url est défini par le chemin relatif du site par rapport à la page externe. Si vous l'utilisez dans un environnement où le chargement d'une page externe à partir d'un domaine différent est autorisé, alors la data-url est définie sur l'URL absolue.

Pages auto-générées et sous-hash d'urls

Certains plugins peuvent choisir de briser dynamiquement le contenu d'une page en pages navigables séparées, qui peuvent ensuite être atteint via des liens profonds. C'est le cas du plugin ListView, qui casse un UL imbriqué (ou OL) dans des pages séparées, qui ont chacunes reçues un attribut data-url afin qu'elles puissent être liées comme n'importe quelle "page" normale dans jQuery Mobile. Toutefois, afin de mettre un lien vers ces pages, la page qui les génère doit d'abord être requêtée à partir du serveur. Pour faire ce travail, les pages qui sont auto-générées par des plugins utilisent la structure spéciale data-url suivante : <div data-url="page.html&subpageidentifier">

Ainsi, par exemple, une page générée par le plugin listview peut avoir un attribut data-url comme ceci : data-url="artists.html&ui-page=listview-1"

Quand une page est demandée, jQuery Mobile sait diviser l'URL à partir de "&ui-page" et faire une requête HTTP avec la portion de l'URL avant la clef. Dans le cas de l'exemple listview mentionné ci-dessus, l'URL devrait ressembler à ceci : http://example.com/artists.html&ui-page=listview-1 ... et jQuery Mobile devrait requêter artists.html, cela générera alors ses sous-pages, la création du div avec data-url="artists.html&ui-page=listview-1", qui affichera alors la page active.

Notez que l'attribut data-url de l'élément contient le chemin de l'URL complet, pas seulement la partie après &ui-page=. Cela permet à jQuery Mobile d'utiliser un mécanisme unique et cohérent qui correspond aux URLs vers la page des attributs data-url.

Les cas où la navigation Ajax ne sera pas utilisée

Sous certaines conditions, les requêtes HTTP normales seront utilisées au lieu des requêtes Ajax. Un cas où cela est vrai, c'est quand un lien pointe vers des pages sur des sites Web externes. Vous pouvez également spécifier qu'une requête http normale doit se faire grâce aux attributs de lien suivants :

  • rel=external

  • target (Avec n'importe quelle valeur, tels que "_blank")

Using the Application Cache

When using the application cache with jQuery Mobile there is at least one important issue to consider. Some browsers, when making requests to the cache will report an http status of 0 on success. This causes jQuery Core's $.ajax to trigger error handlers. The suggested workaround for users leveraging the application cache is to use a jQuery ajax pre-filter. Something like the following (credit to jammus for the snippet):



$.ajaxPrefilter( function(options, originalOptions, jqXHR) {
	if ( applicationCache &&
		 applicationCache.status != applicationCache.UNCACHED &&
		 applicationCache.status != applicationCache.OBSOLETE ) {
		 // the important bit
		 options.isLocal = true;
	}
});

Setting isLocal to true for your ajax requests will alert jQuery Core that it should handle the 0 return values differently. Local requests exhibit similar behavior (ie 0 statuses), and Core will then fall back to determining success based on the presence of content in the xhr responseText attribute.

One important issue to note with the above is that it will set isLocal to true for all requests made via ajax regardless of whether they are in the manifest or not so long as the cache is valid. This works for now because Core only consults the isLocal value when the status is in fact 0 which doesn't affect uncached results. There is no long term guarantee that isLocal will remain isolated in its purpose for handling 0 status values. If that changes it may break your application.

Known limitations

The non-standard environment created by jQuery Mobile's page navigation model introduces some conditions for which you should be aware when building pages:

  • When linking to directories, without a filename url, (such as href="typesofcats/" instead of href="typesofcats/index.html"), you must provide a trailing slash. C'est parce que jQuery Mobile suppose que la section après le dernier caractère "/" dans une URL est un nom de fichier, et il va supprimer cette section lors de la création des urls de base à partir duquel les futures pages seront référencées.

  • Documents loaded via Ajax will select the first page in the DOM of that document to be loaded as a JQM page element. As a result the developer must make sure to manage the ID attributes of the loaded page and child elements to prevent confusion when manipulating the DOM.

  • Any unique assets referenced by pages in a jQuery Mobile-driven site should be placed inside the "page" element (the element with a data-role attribute of "page"). For example, links to styles and scripts that are specific to a particular page can be referenced inside that div. However, a better approach is to use jQuery Mobile's page events to trigger specific scripting when certain pages load. Note: you can return a page from the server with a data-url already specified in the markup, and jQuery Mobile will use that for the hash update. This allows you to ensure directory paths resolve with a trailing slash and will therefore be used in the base url path for future requests.

  • Conversely, any non-unique assets (those used site-wide) should be referenced in the <head> section of an HTML document, or at the very least, outside of the "page" element, to prevent running scripts more than once.

  • The "ui-page" key name used in sub-hash url references can be set to any value you'd like, so as to blend into your URL structure. This value is stored in jQuery.mobile.subPageUrlKey.

  • When traveling back to a previously loaded jQuery Mobile document from an external or internal document with the push state plugin enabled, some browsers load and trigger the popstate event on the wrong document or for the wrong reasons (two edge cases recorded so far). If you are regularly linking to external documents and find the application behaving erratically try disabling pushstate support.

  • jQuery Mobile does not support query parameter passing to internal/embedded pages but there are two plugins that you can add to your project to support this feature. There is a lightweight page params plugin and a more fully featured jQuery Mobile router plugin for use with backbone.js or spine.js.

  • Since we use the URL hash to preserve Back button behavior, using page anchors to jump down to a position on the page isn't supported by using the traditional anchor link (#foo). Use the silentScroll method to scroll to a particular Y position without triggering scroll event listeners. You can pass in a yPos arguments to scroll to that Y location.
